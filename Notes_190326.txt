Here's why each one exists:
django — the core web framework. Everything runs on top of this. Views, models, ORM, admin — all Django.
djangorestframework — adds DRF on top of Django. Gives you APIView, serializers, Response, and JSON handling. Without this, you'd be writing raw JsonResponse everywhere and manually validating input.
celery — the task queue library itself. This is what gives you @shared_task, .delay(), and the worker process. Django alone has no concept of background tasks.
redis — the Python client that lets Celery talk to your Redis server. Celery uses Redis as its broker (the queue), but it needs this library to actually connect to it. Think of Redis the server as the road, and this package as the vehicle.
django-celery-results — stores Celery task results in your PostgreSQL database instead of losing them when Redis restarts. This is what makes CELERY_RESULT_BACKEND = 'django-db' work. Without it, you'd have no persistence of task outcomes.
psycopg2-binary — the PostgreSQL adapter for Python. Django's ORM speaks Python, but PostgreSQL speaks its own protocol. This package is the translator between them. The -binary suffix means it comes pre-compiled — no build tools needed. In production you'd use psycopg2 (non-binary) for better performance, but binary is fine for development.
python-decouple — reads your .env file and injects values into settings.py via config('SECRET_KEY'). Without this, you'd either hardcode secrets (security risk) or write your own env-parsing logic.
django-celery-beat — adds a scheduler on top of Celery. This is what will power your periodic tasks on Day 2 — things like "clean up jobs older than 7 days" running automatically every night. Without this, Celery can only run tasks when explicitly triggered.
locust — the load testing library. You'll write your Locust script on Day 3 to simulate 50, 100, 200 concurrent users hitting your API. This is your performance engineering angle — the thing that separates your project from every other CRUD portfolio project.
